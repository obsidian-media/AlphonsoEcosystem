# Alphonso — Release & Auto-Updater Guide

## Overview

Alphonso uses [Tauri's built-in updater](https://tauri.app/plugin/updater/) with ed25519 signing. When a user runs an installed copy of Alphonso, it checks `https://github.com/Thatisshayan/AlphonsoEcosystem/releases/latest/download/latest.json` on launch and prompts the user to install the new version if one is available.

The release pipeline is fully automated via `.github/workflows/release.yml`. The only manual step is adding two GitHub Secrets — which only needs to be done once.

## Step 1 — Find the Private Key Value

The private key was generated by `npm run updater:keygen` and saved to `.tauri/alphonso-updater.key` on the machine where it was run. This file is gitignored and never committed.

To get the value needed for the GitHub Secret, open `.tauri/alphonso-updater.key` on that machine and copy the entire file contents. It will look like:

```
untrusted comment: minisign secret key: <hex>
<base64-encoded key data>
<base64-encoded key data>
```

Copy everything — all lines including the comment header.

If the file is missing (e.g. the machine is gone), regenerate a new keypair:

```bash
npm run updater:keygen
```

Then update `tauri.conf.json` with the new public key:

```bash
npm run updater:setup
```

> **Warning:** Regenerating the keypair invalidates the public key in `tauri.conf.json`. Existing installs will no longer be able to verify update signatures. Publish a new build with the new public key before releasing any updates signed with the new private key.

## Step 2 — Add GitHub Secrets

Go to: `https://github.com/Thatisshayan/AlphonsoEcosystem/settings/secrets/actions`

Add the following two repository secrets:

### `TAURI_SIGNING_PRIVATE_KEY`

- **Value:** The full contents of `.tauri/alphonso-updater.key` (all lines, including the comment header)
- **Type:** Secret (not environment-scoped — use "Repository secrets")

### `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`

- **Value:** The password entered when the keypair was generated. If no password was set during generation, add this secret with an **empty value** (GitHub allows empty secrets — just leave the value field blank and click Add)
- **Type:** Secret

Both secrets are consumed by the `release.yml` workflow step:

```yaml
- name: Build signed Tauri app
  env:
    TAURI_SIGNING_PRIVATE_KEY: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY }}
    TAURI_SIGNING_PRIVATE_KEY_PASSWORD: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY_PASSWORD }}
  run: npx.cmd tauri build
```

## Step 3 — Trigger a Release

### Via Git Tag (recommended)

```bash
# Bump version in tauri.conf.json and Cargo.toml first, then:
git tag v2.0.1
git push origin v2.0.1
```

The `release.yml` workflow fires automatically on any `v*` tag.

### Via Manual Dispatch

Go to: `Actions → Release → Run workflow`

Enter the version tag (e.g. `v2.0.1`) in the input field and click Run.

## What the Release Workflow Does

1. Runs `npm run verify:app` (lint + test + build) — release is aborted if any step fails
2. Runs `cargo test` and `cargo clippy -- -D warnings`
3. Runs `npx.cmd tauri build` with the signing key injected — produces `Alphonso_X.Y.Z_x64-setup.exe` and `Alphonso_X.Y.Z_x64-setup.exe.sig`
4. Generates `latest.json` — the updater manifest with version, pub_date, download URL, and ed25519 signature
5. Creates a GitHub Release with the installer, signature file, and `latest.json` as release assets

The updater endpoint in `tauri.conf.json` points to:
```
https://github.com/Thatisshayan/AlphonsoEcosystem/releases/latest/download/latest.json
```

GitHub's `/releases/latest/download/` path always serves the most recent non-prerelease's assets, so no manifest hosting infrastructure is needed.

## Step 4 — Verify

After the release workflow completes:

```bash
# From the project root (not required to have the private key locally)
npm run updater:verify
```

This script checks:
- `tauri.conf.json` has `pubkey` configured
- The `latest.json` manifest is reachable at the GitHub endpoint
- The manifest structure is valid

You can also run it before a release to confirm the pipeline is ready:

```
✓ tauri.conf.json pubkey: configured
✓ Updater endpoint: reachable
✓ Manifest version: 2.0.0
✓ Manifest platform: windows-x86_64
```

## Troubleshooting

### Build fails with "TAURI_SIGNING_PRIVATE_KEY is required"

The secret is missing or named incorrectly. Check `Settings → Secrets → Actions` and confirm both secrets exist with the exact names `TAURI_SIGNING_PRIVATE_KEY` and `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`.

### Installed app shows no update available

- Confirm the `latest.json` was uploaded as a release asset (not as a pre-release)
- Confirm the `version` field in `latest.json` is higher than the installed version
- The updater checks version using semver — `2.0.1 > 2.0.0` is required for the prompt to appear

### Signature verification fails on install

The signing keypair was regenerated after the `pubkey` in `tauri.conf.json` was last published. Publish a new build with the updated `pubkey` before distributing updates signed with the new key.

### `npm run updater:verify` shows "setup_required" for the private key

This is expected on CI and on any machine that does not have `.tauri/alphonso-updater.key`. The verify script checks local file presence as a convenience — the actual signing key is injected at build time via the `TAURI_SIGNING_PRIVATE_KEY` environment variable, not via the local file path.
